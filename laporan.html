4. Latar Belakang

Banyak sekolah masih menggunakan sistem yang terpisah-pisah untuk mengelola jurnal mengajar, data siswa, data BK, absensi, dan laporan akademik. Setiap bagian menggunakan aplikasi atau catatan masing-masing tanpa integrasi data. Akibatnya sering terjadi data ganda, data tidak sinkron antar bagian, proses administrasi yang lambat, serta sulitnya membuat laporan terpadu.

Contoh nyata: data siswa yang sudah dicatat di bagian kesiswaan harus dimasukkan ulang oleh guru BK atau guru jurnal mengajar. Hal ini tidak efisien dan rawan kesalahan. Data siswa antara satu bagian dengan bagian lain bisa berbeda karena tidak ada integrasi database.

Oleh karena itu, kami merancang dan mengembangkan Web Sekolah Terintegrasi. Sistem ini menggabungkan beberapa modul penting sekolah dalam satu platform dengan satu database utama. Data siswa, guru, kelas, dan mata pelajaran cukup disimpan satu kali dan dapat digunakan bersama oleh semua modul. Sistem juga dirancang dengan prinsip Scalable System Design agar dapat berkembang seiring pertumbuhan jumlah siswa, guru, dan akses pengguna.

5. Tujuan Proyek

  1. Merancang dan mengembangkan web sekolah terintegrasi yang terdiri dari 5 modul utama (Manajemen Pengguna, Kesiswaan, Jurnal Mengajar, BK, Absensi).
  2. Menerapkan satu database utama yang digunakan bersama oleh seluruh modul sehingga data konsisten dan tidak terduplikasi.
  3. Menerapkan konsep Scalable System Design dalam arsitektur sistem, pembagian vCPU, strategi scaling, load balancing, caching, monitoring, dan logging.
  4. Merancang hak akses pengguna berdasarkan role (Admin, Kepala Sekolah, Guru, Guru BK, Wali Kelas, Siswa, Orang Tua).
  5. Membuat dokumentasi proyek yang sistematis dan mudah dipahami.
  6. Mengunggah source code ke GitHub dan membuat video presentasi di YouTube.

6. Deskripsi Sistem

Web Sekolah Terintegrasi adalah sistem informasi berbasis web yang dibangun menggunakan Node.js Express.js sebagai backend, MySQL sebagai database, Bootstrap 5 sebagai framework frontend, dan EJS sebagai template engine. Sistem menggunakan pendekatan Modular Monolith Architecture di mana setiap modul memiliki router dan view sendiri tetapi berjalan dalam satu aplikasi.

Sistem mendukung 7 role pengguna dengan hak akses berbeda. Setiap modul dilindungi oleh middleware autentikasi dan RBAC (Role-Based Access Control). Semua aktivitas pengguna dicatat dalam tabel activity_logs untuk audit trail.

9. Arsitektur Sistem

Sistem menggunakan pendekatan Modular Monolith Architecture. Seluruh modul berjalan dalam satu aplikasi Express.js, tetapi setiap modul dipisahkan dalam direktori dan router masing-masing. Hal ini memudahkan pengembangan dan pemeliharaan, serta membuka jalan untuk pemisahan menjadi service terpisah di masa depan.

  • Penggunaan ENUM untuk kolom dengan nilai tetap (status, gender, day_of_week)

13. Alur Kerja Sistem

  1. Login: Pengguna membuka http://localhost:3000, memasukkan username dan password. Sistem memverifikasi kredensial dan membuat session.
  2. Dashboard: Setelah login, pengguna diarahkan ke dashboard yang menampilkan statistik, grafik, dan log aktivitas sesuai rolenya.
  3. Navigasi: Pengguna memilih modul melalui navbar. Menu yang tampil disesuaikan dengan role.
  4. Operasi Data: Pengguna dapat menambah, mengedit, menghapus data sesuai hak akses. Setiap perubahan dicatat ke activity_logs.
  5. Logout: Session dihapus, pengguna kembali ke halaman login.

15. Penjelasan Unsur Scalable System Design

15.1 Modular Architecture

Sistem dibagi menjadi 5 modul terpisah (Pengguna, Kesiswaan, Jurnal, BK, Absensi). Setiap modul memiliki direktori, router, dan view sendiri. Jika ingin menambah modul baru (misal: modul Nilai), cukup buat folder baru di src/modules/, daftarkan route di app.js, dan tambahkan menu di navbar.

15.2 Centralized Database

Semua modul menggunakan satu database MySQL (db_sekolah_ssd). Data siswa cukup disimpan sekali di tabel students dan dapat digunakan oleh semua modul. Tidak ada duplikasi data, konsistensi terjaga.

15.3 Load Balancing

Load balancer (vCPU 5) mendistribusikan request pengguna ke server yang tepat. Jika banyak guru mengakses jurnal mengajar secara bersamaan, request dapat dibagi ke beberapa instance server.

15.4 Horizontal Scaling

Jika modul Jurnal Mengajar mengalami lonjakan akses, kita dapat menambah instance server untuk modul tersebut tanpa mengganggu modul lain. Arsitektur modular memungkinkan scaling per-modul.

15.5 Vertical Scaling

Server database dapat ditingkatkan kapasitas CPU, RAM, atau storage untuk menangani data yang semakin besar. MySQL dapat dikonfigurasi dengan buffer pool yang lebih besar untuk performa query.

15.6 API-Based Integration

Setiap modul menggunakan data bersama melalui query ke database yang sama. Di masa depan, modul dapat dipisah menjadi service mandiri yang berkomunikasi melalui REST API.

15.7 Role-Based Access Control (RBAC)

Middleware allowRoles() membatasi akses berdasarkan role. Guru tidak bisa mengakses data BK, siswa hanya bisa melihat dashboard, dan admin memiliki akses penuh.

15.8 Database Optimization

Index pada kolom student_id, teacher_id, class_id, subject_id, dan attendance_date mempercepat pencarian. Foreign key constraints menjaga integritas data. Normalisasi hingga 3NF mencegah redundansi.

15.9 Caching

Rencana implementasi Redis untuk cache data yang sering diakses: daftar kelas, guru, mapel, dan jadwal pelajaran. Cache mengurangi beban query ke database.

15.10 Monitoring & Logging

Tabel activity_logs mencatat semua aktivitas pengguna (login, create, update, delete). Dashboard menampilkan log aktivitas terbaru. Monitoring performa server (CPU, RAM, request rate) direncanakan menggunakan tools seperti PM2 atau Grafana.

17. Jawaban Pertanyaan Analisis (Bagian M)

1. Mengapa sistem web sekolah ini membutuhkan desain yang scalable?

Karena sekolah akan terus berkembang. Jumlah siswa bertambah setiap tahun, jumlah guru bertambah, dan akses ke sistem meningkat terutama pada jam-jam sibuk (saat guru mengisi jurnal atau absensi). Tampa desain scalable, sistem akan lambat, error, atau bahkan down saat beban tinggi. Dalam proyek ini, skalabilitas diterapkan melalui pembagian vCPU, modular architecture, dan rencana caching.

2. Apa masalah yang mungkin terjadi jika semua modul dibuat dalam satu aplikasi tanpa pembagian layanan?

Masalah utama: (1) Jika satu modul error, seluruh sistem bisa terganggu. (2) Sulit melakukan scaling karena semua modul terikat dalam satu kode. (3) Deployment menjadi berat karena harus deploy seluruh aplikasi meskipun hanya mengubah satu modul. (4) Resource usage tidak efisien karena modul dengan beban rendah ikut menggunakan resource yang sama dengan modul beban tinggi.

3. Bagaimana cara sistem menjaga konsistensi data jika semua modul menggunakan satu database?

Dengan menggunakan foreign key constraints, transaction (BEGIN/COMMIT) untuk operasi multi-tabel, dan validasi data di sisi server sebelum insert/update. Contoh: saat input absensi, student_id harus valid (foreign key ke tabel students), sehingga tidak ada absensi untuk siswa yang tidak terdaftar.

4. Modul mana yang paling mungkin mengalami beban akses paling tinggi? Jelaskan alasannya.

Modul Jurnal Mengajar. Setiap hari, puluhan hingga ratusan guru login untuk mengisi jurnal pembelajaran. Intensitas akses tinggi terjadi pada jam-jam tertentu (pagi hari sebelum mengajar, siang setelah mengajar). Modul Absensi juga berpotensi tinggi karena diakses setiap hari oleh guru untuk mencatat kehadiran.

5. Bagaimana strategi scaling yang tepat jika jumlah pengguna meningkat?

Strategi horizontal scaling: menambah instance server untuk modul dengan beban tinggi (misal: Jurnal Mengajar dari 1 menjadi 2 server). Vertical scaling: meningkatkan spesifikasi server database (RAM, CPU). Kombinasi keduanya (hybrid scaling) adalah strategi paling optimal.

6. Apa risiko dari penggunaan satu database terpusat?

Risiko utama: (1) Single point of failure — jika database down, seluruh sistem lumpuh. (2) Bottleneck — semua query dari semua modul terpusat ke satu database. (3) Ukuran database membesar seiring waktu, memperlambat query. Solusi: replication (master-slave), caching, dan partitioning.

7. Bagaimana cara mengurangi risiko database menjadi bottleneck?

(1) Implementasi Redis cache untuk data yang jarang berubah (daftar kelas, guru, mapel). (2) Query optimization dengan index yang tepat. (3) Connection pooling menggunakan mysql2 pool. (4) Read-replica untuk query SELECT yang berat. (5) Database partitioning jika data sudah sangat besar.

8. Bagaimana sistem membatasi akses data sensitif, terutama data BK?

Menggunakan middleware RBAC allowRoles('guru_bk', 'admin', 'kepala_sekolah') pada route BK. Pengguna dengan role lain (guru biasa, siswa) akan diarahkan ke dashboard atau mendapat pesan error jika mencoba mengakses /bk. Data konseling, pelanggaran, dan prestasi hanya bisa diinput dan dilihat oleh guru BK dan admin.

9. Bagaimana sistem tetap dapat dikembangkan jika sekolah ingin menambah modul baru?

Karena menggunakan Modular Architecture, menambah modul baru sangat mudah: (1) Buat folder baru di src/modules/nama-modul/ dengan routes.js. (2) Buat folder view di src/views/nama-modul/. (3) Daftarkan route di src/app.js. (4) Tambahkan menu di navbar main.ejs. (5) Jika perlu tabel baru, tambahkan di schema.sql. Semua tanpa mengubah kode modul yang sudah ada.

10. Bagaimana monitoring dan logging membantu menjaga kestabilan sistem?

Tabel activity_logs mencatat setiap aktivitas pengguna (login, input data, edit, delete). Log ini berguna untuk: (1) Mendeteksi aktivitas mencurigakan (misal: banyak gagal login). (2) Audit trail untuk mengetahui siapa yang mengubah data. (3) Troubleshooting error. (4) Analisis penggunaan sistem (modul mana yang paling sering diakses). Dashboard menampilkan 10 aktivitas terbaru untuk monitoring real-time.

18. Kesimpulan

Proyek Web Sekolah Terintegrasi ini berhasil mengembangkan sistem informasi sekolah yang terdiri dari 5 modul utama (Manajemen Pengguna, Data Kesiswaan, Jurnal Mengajar, BK, dan Absensi) dengan satu database terpusat dan sistem hak akses berbasis role. Sistem dibangun menggunakan Node.js Express.js, MySQL, Bootstrap 5, dan EJS.

Penerapan konsep Scalable System Design terlihat pada:

  • Modular Architecture yang memudahkan pengembangan dan scaling
  • Centralized Database dengan 16 tabel terelasi dan optimasi index
  • Pembagian 6 vCPU dengan alasan teknis yang jelas
  • RBAC dengan 7 role pengguna
  • Monitoring dan logging melalui activity_logs
  • Rencana implementasi caching, load balancing, dan horizontal/vertical scaling

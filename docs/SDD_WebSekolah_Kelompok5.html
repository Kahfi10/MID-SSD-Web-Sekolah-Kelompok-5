1. Pendahuluan

1.1 Tujuan Dokumen

Dokumen Software Design Document (SDD) ini mendeskripsikan desain teknis sistem Web Sekolah Terintegrasi. Dokumen ini mencakup arsitektur sistem, desain perangkat lunak (use case, class diagram, sequence diagram), desain database, desain antarmuka pengguna, dan matriks ketertelusuran kebutuhan.

SDD ini merupakan turunan langsung dari dokumen SRS v1.0 dan menjadi acuan implementasi bagi tim pengembang. Dokumen mengacu pada standar IEEE 1016-1998.

1.2 Ruang Lingkup

SDD ini mencakup desain untuk 5 modul utama sistem: Autentikasi, Manajemen Pengguna, Kesiswaan, Jurnal Mengajar, Bimbingan Konseling, dan Absensi Siswa beserta fitur lintas modul (Dashboard, Profil, Pagination, Search).

2. Arsitektur Sistem

2.1 Gambaran Arsitektur

Sistem menggunakan pendekatan Modular Monolith Architecture. Seluruh modul berjalan dalam satu proses Node.js Express.js, namun setiap modul dipisahkan dalam direktori dan router masing-masing. Pendekatan ini dipilih karena memudahkan pengembangan awal sekaligus membuka jalan migrasi ke microservices di masa depan.

Diagram Arsitektur Sistem:
Lihat docs/arsitektur.jpg di repository GitHub.
Menampilkan 3 layer: Client Layer → Application Layer → Data Layer beserta 6 vCPU.

2.4 Alur Request-Response

Berikut alur umum setiap request dari browser pengguna hingga response dikembalikan:

  1. Browser mengirim HTTP/HTTPS request ke URL (misal: POST /bk/cases/store)
  2. Nginx menerima request di port 443, melakukan SSL termination, dan meneruskan ke port 3000
  3. Express.js mencocokkan route dengan router yang terdaftar di app.js
  4. Middleware isAuthenticated memeriksa session — jika tidak ada, redirect ke /auth/login
  5. Middleware allowRoles() memeriksa role pengguna — jika tidak sesuai, redirect ke /dashboard
  6. Route Handler memproses logika bisnis, melakukan query ke MySQL via connection pool
  7. MySQL mengembalikan data hasil query
  8. Route Handler merender view EJS dengan data yang diterima, atau redirect dengan flash message
  9. Browser menerima HTML response dan merender tampilan

2.5 Strategi Scaling

Jenis ScalingTargetImplementasiKapan Digunakan
Horizontal ScalingvCPU 1 & 3Tambah instance Node.js di belakang Nginx load balancer (round-robin)Saat jumlah pengguna meningkat signifikan
Vertical ScalingvCPU 2Tingkatkan spesifikasi server database (RAM, CPU, storage)Saat data semakin besar dan query melambat
CachingvCPU 6Redis cache untuk data statis: daftar kelas, mapel, guru, rolesImplementasi fase berikutnya
Read ReplicavCPU 2MySQL master-slave replication untuk query SELECTSaat read traffic jauh lebih tinggi dari write

4.4 Optimasi Database

  • Index: Dibuat pada kolom yang sering di-query: student_id, teacher_id, class_id, attendance_date, created_at
  • Normalisasi: Database dinormalisasi hingga 3NF — tidak ada data redundan, setiap atribut bergantung penuh pada primary key
  • ENUM: Kolom dengan nilai tetap menggunakan ENUM: gender, status (students), status (bk_cases), status (attendances), day_of_week
  • Connection Pool: mysql2/promise pool dengan connectionLimit: 10 untuk efisiensi koneksi
  • Parameterized Query: Semua query menggunakan placeholder ? untuk mencegah SQL Injection
